> 用途：保存可复现证据，并整理面试时能讲清楚的决策过程。计划、事实和结果必须分开记录。

## 1. 目标与成功标准

- 任务：用 GRPO 微调 `Qwen/Qwen3.5-9B-Base`，让模型可多轮检索 `/data/docs` 后回答技术培训题。
- 主指标：`validation/accuracy`，以平台记录的最佳值为准。
- 资源：单张 H200 141GB，同时最多一个作业；实际可用时间约 30 小时。
- 硬约束：不得修改 `/data/datasets/qa_rl`；最终答案必须含 `\boxed{...}`。
- 提交物：GitHub Fork、最佳作业 ID、指标截图、实验说明。

## 3. 设计决策

### 3.1 最小变量

保留单轮 QA 的数据格式、答案奖励、模型、LoRA 和 GRPO 主结构，只新增两件事：

1. 多轮协议：`<search>关键词</search>` 或 `<answer>...\boxed{答案}</answer>`。
2. 本地 Markdown 检索环境：读取 `/data/docs`，返回固定数量、固定长度的相关片段。

这样可以把性能变化主要归因于“是否能检索资料”，也减少接口风险。

### 3.2 检索实现

- 使用纯 Python、确定性的 BM25 风格稀疏检索。
- 中文采用字符二元组，英文和数字采用词项；不依赖联网模型或额外向量服务。
- 文档启动时只索引一次；按 Markdown 标题和段落切块。
- 每次最多返回 3 个片段，并限制总字符数，避免环境反馈挤满上下文。

### 3.3 奖励设计

- 有效检索轮奖励为 0。
- 最终答案复用现有 QA 奖励；没有 `\boxed{}` 仍为格式罚分。
- 不给“调用搜索”额外正奖励，因为平台把总奖励均值直接命名为 `validation/accuracy`；搜索奖励会污染主指标并诱导无意义检索。

### 3.4 预算策略

只允许串行作业，不做网格搜索。每轮先写假设和停止条件，再提交：

1. 本地测试和 `lab validate`。
2. 极短冒烟作业：验证数据挂载、文档索引、多轮状态机和指标上报。
3. 正式 v1：只采用一个稳定配置。
4. 仅根据验证轨迹修复最主要失败模式，再决定是否提交 v2。
5. 预留最后 3 小时做最终验证、截图、Git 和说明。

## 4. 已遇到的困难与解决办法

| 困难 | 证据 | 处理 |
| --- | --- | --- |
| 浏览器最初没有平台登录态 | 打开后进入 `/login` | 让用户自行登录，随后只读确认配额与页面结构 |
| 手册写 48 小时，用户确认只剩约 30 小时 | 用户补充 | 以当前通知为准，压缩为串行的“冒烟 → 正式 → 一次定向修正” |
| 手册、仓库中的卡型描述不完全一致 | 旧文档提到 H100，控制台显示 H200 | 以控制台实时配额和实验 `cluster=h200` 为准 |
| README 引用了不存在的 QA Agent 实验 | 当前 Git tree 无对应目录 | 复用单轮 QA 判分，并按 NeMo-RL v0.6.0 官方多轮接口实现 |
| NeMo-RL 自带 RAG 环境会单轮终止，且默认依赖英文 BERT tokenizer | v0.6.0 `retriever.py` | 只借鉴接口，不直接使用；实现本地中文稀疏检索和显式状态机 |
| 多轮检索结果容易造成序列截断 | `run_multi_turn_rollout` 会在超过总长度时标记 truncated | 限制检索条数与字符数，监控 `truncation_rate`，不用 `</think>` 截断最终答案 |
| 只有一个并发作业 | 控制台配额 `0/1` | 不并行扫参；运行中采用低频定时检查 |
| `uv sync` 下载依赖较慢，后续检查都在等待安装锁 | 三个 `uv run` 同时无输出，进程状态显示均等待同一安装 | 取消重复等待，只保留一次安装；先用系统现有 `pytest` 和 `py_compile` 验证纯 Python 核心 |

## 7. 面试讲述骨架

1. 我先把目标拆成两个可验证问题：模型能否稳定完成多轮协议；检索内容是否真的帮助最终答案。
2. 我没有重写已有判分，而是复用单轮 QA 的五题型奖励，只新增检索状态机，控制变量并降低实现风险。
3. 我发现官方 RAG 环境不适合直接使用：它单轮终止，并依赖英文 tokenizer；因此实现了无外部依赖的中文 BM25 风格检索。
4. 我刻意不给检索动作奖励，因为平台的 accuracy 是总奖励均值，额外奖励会污染指标并诱导刷搜索次数。
5. 单 GPU、约 30 小时下，我用冒烟作业先验证真实链路，再依据验证样本做一次定向修改，而不是盲目网格搜索。
6. 最终讲述必须补上真实作业 ID、最佳准确率、失败轨迹和对应修正；在跑出结果前不提前声称提升。
